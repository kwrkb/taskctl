# VISION.md  taskctl

## 一行で

Windows タスクスケジューラで **タスクが失敗した / 失敗しそうな理由を特定し、「次に何をやればいいか」を具体的に示す** 診断ツール。設定は自分では変更しない。出力は**日本語英語**に対応する。

コマンド名は暫定的に `taskctl`。中核サブコマンドは `doctor`。

---

## このツールの主張

タスクスケジューラ周りで「何が悪いか」を出す手段は既にある（標準の結果コード表示、Operational ログ、各種 GUI）。だが **「だから次にこれをやれ」まで言ってくれるものが無い。**

- 最終結果は `0x41303` のような生コードで表示され、意味は自分で調べる
- しかもそのコードが「プログラムの終了コード」なのか「タスクスケジューラの状態コード」なのか区別されない
- 相対パス作業ディレクトリ未設定プロファイル依存といった「失敗しやすい設定」を事前に指摘してくれない

**本ツールの価値は、この翻訳と次アクションの提示に集約する。** それ以外はやらない。

---

## 作らない範囲（標準機能で足りる）

運用ループはすべて標準 PowerShell で完結する。ここは対象外とし、車輪を再発明しない。

- 一覧状態確認 : `Get-ScheduledTask` / `Get-ScheduledTaskInfo`
- 設定の忠実な取得（XML） : `Export-ScheduledTask`
- 登録更新 : `Register-ScheduledTask`
- 有効化 / 無効化 / 手動実行 : `Enable-` / `Disable-` / `Start-ScheduledTask`
- 履歴 : `Get-WinEvent -LogName Microsoft-Windows-TaskScheduler/Operational`

config-as-code でのデプロイ（Ansible / Azure DevOps / GitHub Actions ＋ XML）も既存の領域であり、対象外。

**本ツールが埋める穴は「結果コードの翻訳」「設定ミスの検出」「次アクションの提示」の3点のみ。**

---

## 中核コンセプト：何をやればいいか分かる

diagnostic の出力は「何が悪いか」で終わらせず、**必ず次の一手を添える。** ただし、外れた次アクションは何も言わないより有害なので、**確信度でランク分けし、断定できる時だけ断定する。**

次の一手は3種類。VISION 上の重大度（Error / Warning / Notice）に対応させる。

- **修正**（Error高確信）：原因がほぼ確定している。コピペできる標準コマンドをそのまま提示する。
- **調査**（Warning中確信）：断定できない。まず真因を掴むための情報収集コマンド（手動実行、Operational ログ確認など）を提示する。
- **判断**（Notice仕様かもしれない）：ツールには決められない。「これは意図通りか？」と問いを返すだけにする（例：AC 電源時のみ実行）。

そして **自動では直さない。** doctor はコマンドを *見せる* だけで、実行するのはユーザー。これにより read-only を保ち、「設定を自動変更しない」を既定方針として貫く。「次にやることが分かる」と「勝手に壊さない」を両立させる肝がここ。

## 診断出力の型

1件の所見は、必ず次の3点セットで構成する。人間が読む説明文（下記の日本語部分）は表示言語に応じて日英で切り替わるが、**コマンドと機械識別子は言語に依存しない**（後述の i18n を参照）。

```
これは何   : コード/設定の意味を、状態コードか終了コードかを区別して説明
考えられる原因 : 断定せず候補を示す
次の一手 [ランク] : 修正 / 調査 / 判断 のいずれか。コマンドは実行せず提示のみ
```

例（失敗タスク日本語表示）:

```
 MyBackup   最終結果: 0x2 (10進 2)   [ERROR_FILE_NOT_FOUND]

これは何:
  実行しようとしたファイルが見つからない、という Windows システムエラー。
  プログラム自身の終了コードではありません。

考えられる原因:
  操作のパスが存在しない、またはタスク実行時の文脈（別ユーザー
  マップドライブ）から解決できない。
    操作: powershell.exe -File Z:\scripts\backup.ps1

次の一手 [調査]:
  Z: がネットワークドライブなら、タスク実行時に未マウントの可能性。
  共有(UNC)上の実行は cmd 経由が有効（公式回避策）:
    cmd /c \\server\share\backup.ps1
```

**ツールの本体は、この型で埋めた「結果コード翻訳表」と「検出ルール集」に他ならない。**

## 表示言語（日英 i18n）

CLI の i18n には固有の落とし穴がある。「なんとなく難しい」の正体はこの3つ（翻訳境界ロケール決定エンコーディング）で、設計で先に潰す。

### 1. 翻訳する層を厳密に分ける

localize するのは **人間向けの説明文だけ**（これは何 / 原因 / 次の一手の説明テキスト）。次は **絶対に翻訳しない**：

- 実行提示するコマンド（翻訳したら壊れる）
- 定数名（`SCHED_S_TASK_READY` 等。grep検索の安定キー）
- コード値（`0x` / 10進）、`kind` / `severity` などの機械識別子

この境界を破ると「コマンドが動かない」「ログが grep できない」という実害になる。**翻訳されるのはプロース層のみ**、が鉄則。

### 2. データ構造：言語非依存レジストリ ＋ ロケール別カタログ

中核データを2層に分ける。

- **レジストリ**（1つ言語非依存）：`code  constant / kind / severity / is_failure`。「コードについての事実」。
- **メッセージカタログ**（ロケールごと：`ja` / `en`）：`code  { meaning, cause, next.text }`。「その言い方」。

こうすると「どのコードが存在し、失敗か否か」という判定ロジックは1箇所に留まり、翻訳はカタログを差し替えるだけになる。**カタログのキー集合 ＝ レジストリのキー集合** を coverage テストで担保する（`ja` に有って `en` に無い、を検出）。未対応の所は英語へフォールバックし、空文字は絶対に出さない。

このカタログは **engine の実装言語にも依存しない**。PowerShell 版 doctor も、将来の Go 版 doctor も、同じ `ja` / `en` カタログを読む。i18n の作り込みが、そのまま多言語実装のコストも下げる。

> 補足：現行の `result-codes.yaml` は事実と日本語プロースが同居している。i18n 対応では、これをレジストリ（neutral）＋ `messages/ja.yaml` `messages/en.yaml` に分割するのが最初のリファクタになる。

### 3. ロケールの決定（Windows CLI の現実）

ブラウザの `Accept-Language` は無い。優先順位を決め打ちする。

1. `--lang ja|en`（明示フラグ）
2. 環境変数（例 `TASKCTL_LANG`）
3. OS の UI カルチャ（`$PSUICulture` / `GetUserDefaultUILanguage`）
4. 既定 = `en`

### 4. 文字エンコーディング（CLI 最大の課題）

日本語をコンソールに出す時の定番の罠に正面から対処する。

- 出力は **UTF-8 を既定**とする。ただし Windows コンソールは既定コードページ（日本語環境で CP932）や PowerShell 5.1 の出力エンコーディングで化けうる。検出できない場合は、化けた時の対処（`chcp 65001` / PowerShell 7 の利用）を案内する。
- **`--json` 出力は表示ロケールに関わらず常に UTF-8**。機械可読を壊さない。

### 5. --json と i18n

JSON には言語非依存フィールド（`code` / `constant` / `kind` / `severity` / `is_failure` / `message_key`）を必ず含め、加えて現在ロケールの `message` / `action` を載せる。消費側は key で自前ローカライズも、提供テキストの利用も選べる。

## 診断項目

### 結果コードの翻訳

Windows の結果コードを、種類を区別して説明する。

- タスクスケジューラの状態コード（`SCHED_S_*` / `0x4130x` 系。例：まだ実行されていない実行中無効。**これらは失敗ではなく状態**であり、プロセス終了コードと混同されやすい）
- Windows システムエラー（`0x2` = ファイルなし、`0x80070005` = アクセス拒否 等）
- 実行プログラムの終了コード / PowerShell の終了コード / ユーザー定義の終了コード

不明なコードは「不明」と表示し、16 進10 進の両方を示す。`LastTaskResult` は符号付き 32bit で返るため、正規化してから引く。断定しない。

> 翻訳表の中身（コード  意味）は記憶で埋めず、Microsoft の一次資料で検証してから確定する。ここが誤るとツールの信頼が根本から崩れる。（実装済みの `result-codes.yaml` は `SCHED_S_`/`SCHED_E_` を Microsoft Learn で検証済み。）

### 設定ミスの検出

失敗しやすい設定を検出し、上記の型で次アクションを示す。初期候補：実行ファイル／スクリプト／作業ディレクトリの不在、作業ディレクトリ未設定＋相対パス、ネットワークドライブプロファイル依存、PowerShell/バッチ/WSL の起動指定不備、ログオン中のみ／AC 電源時のみ／アイドル時のみの条件、実行時間制限が短い、多重起動許可、タスク無効、次回実行なし、長期間未実行。

検出ルールも**極力データ（宣言的ルール＋小さな評価器）で表現**する。ロジックをコードに散らすと、i18n も多言語実装も分岐して壊れる。判定は1箇所、プロースはカタログ、が原則。

## 誤検知への規律

信頼が命のツールなので、確信度に正直であることを最優先する。

- **文脈差を前提にする。** taskctl が動く文脈と、タスクが実際に走る文脈（別ユーザーマップドライブ別プロファイル）は異なる。ファイル存在チェックはこの差で誤爆しうるため、原則 Error にせず、この差そのものを「調査」として提示する。
- 相対パス検出引用符判定などヒューリスティックは Warning 止まり。「修正」コマンドは出さず「調査」に留める。
- 低確信の所見に、断定的な「修正」コマンドを出さない。確信がある時だけ手を引く。

---

## 入力とデータ源（すべて read-only）

- 設定 : `Export-ScheduledTask` の XML（忠実ロケール非依存）
- 実行情報 : `Get-ScheduledTaskInfo`（設定とは別取得。相関が必要）
- 履歴（任意） : Operational ログ（既定で無効なことがある前提。無ければその旨を案内）

留意：タスクオブジェクトが持つ実行結果は直近1回分のみ。エンコーディング（CP932 / UTF-16BOM）とトリガー時刻のローカル/TZ は取り扱い時に考慮する（i18n の項も参照）。

## 想定コマンド

最小構成。増やさない方向で考える。

```
taskctl doctor              # 全ユーザータスクを走査。状態一覧＋問題のあるものに診断
taskctl doctor <task>       # 1本を深掘り
taskctl explain <code>      # 結果コード単体を翻訳（例: taskctl explain 0x41303）

共通フラグ:
  --lang ja|en   表示言語（既定は環境/OSから決定、最終フォールバックは en）
  --json         構造化出力（常に UTF-8。言語非依存フィールド＋現在ロケールの文言）
  --verbose      生の設定も表示
```

`doctor` を「タスクの健康診断」と位置づけ、一覧と診断を同じ出力に載せることで、「壊れた時」だけでなく「たまに眺める」用途も兼ねさせる。

## 技術方針

### まず PowerShell、育ったら Go

ツールの本体は「翻訳表＋検出ルール＋メッセージカタログ」という**言語非依存の資産**であり、取得は標準コマンド出力を読むだけで済む。

- **v1 は PowerShell モジュールで始める。** 価値の本体に最短で触れ、COM の複雑さを回避する。
- ルールが増え、単一バイナリ配布とテスト容易性が欲しくなった時点で Go へ移す。データ資産はそのまま移植でき、`ja`/`en` カタログも共有できる。
- 取得層はインターフェースで隔離。診断ルールは正規化モデル上の**純粋関数**にし、実 XML を fixture にすれば Windows 実機なしでテストできる。
- **COMtaskmaster forkwrite(apply) は v1 スコープ外。**

---

## UX 方針

- デフォルト出力は短く、**問題と次の一手を先頭に**。詳細は `--verbose` / `--json`。
- 失敗を示すだけで終わらせない。必ず「調査 / 修正 / 判断」のいずれかを添える。
- 日英いずれの表示でも、コマンドはそのままコピペして動く（＝コマンドは非翻訳）。
- 終了コードで自動化に乗せる（例：`0` 問題なし / `2` 警告あり / `3` 重大な問題あり。詳細は実装時）。

## 対象ユーザー / 対象外

**対象**：PowerShellバッチWSL を定期実行する個人／開発者、バックアップ運用者、そして「失敗したが原因が分からない」人。日本語話者英語話者の双方。

**対象外**：apply / plan / write 全般（標準 `Register-ScheduledTask` に任せる）、COM 実装、独自スケジューラ、常駐サービス、GUI、config-as-code デプロイ、複数 PC 集中管理、秘密情報パスワード管理、Windows 以外の OS。

---

## 成功条件

- 失敗コードの意味を数秒で理解できる（状態コードか終了コードかも分かる）
- 失敗しやすい設定を、失敗する前に指摘できる
- 提示される「次の一手」が具体的（多くはコピペ可能）で、外れが少ない
- **日本語でも英語でも自然に読め、コマンドは非翻訳で正しく動く**
- 自動変更しないため、ツールを使ってタスクを壊すことがない
- Windows 標準機能と共存し、単一バイナリ（or モジュール）を置くだけで使える

---

## 判断基準

機能追加設計判断は以下で測る。

1. 失敗原因が分かりやすくなるか
2. 「次に何をやればいいか」が明確になるか
3. 確信度に正直か（外れた断定をしないか）
4. 自動変更せず、安全に保てるか
5. 標準機能と自然に共存するか
6. 診断資産（翻訳表ルールカタログ）が言語非依存で残るか
7. i18n の境界（翻訳するのはプロース層のみ）を壊していないか

「タスクスケジューラを操作できるか」ではなく、「**失敗したユーザーが、次の一手に迷わなくなるか**」を基準とする。

## 将来像

```
v1  PowerShell モジュール : 翻訳表＋検出ルールを「これは何／原因／次の一手」の型で育てる
                            レジストリ＋ja/enカタログに分離し、日英で出せる状態にする
v2  Go 単一バイナリ       : データ資産とカタログを移植し、配布性とテスト性を得る
その先（任意）            : カタログ追加で対応言語を増やす。write(apply) は本体ではない
```

本ツールの本体は、タスクを操作するコードではない。**失敗したタスクを前に「次に何をすればいいか」を、確信度に正直な形で、読む人の言語で示す知識（翻訳表ルールカタログ）** である。
